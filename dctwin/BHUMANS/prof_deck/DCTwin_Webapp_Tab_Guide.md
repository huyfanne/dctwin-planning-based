## The one-sentence thesis

> By making the 26 C inlet cap a hard, non-negotiable constraint and energy a soft objective, a physics-grounded dual-loop controller can run a 2 MW hall warmer and slower for a measured 3.63% weekly HVAC saving while a learning loop drives its prediction error from 1.33% to 0.10% over six weeks and never once touches the cap.

## How to open

Every data center on earth wastes energy over-cooling, because the one thing an operator cannot afford is a server inlet over its limit, so they buy safety with electricity. Our question was the inverse: how warm and how slow can you run a 2 MW hall before you touch the line, and can you prove you never will? This app is the answer made operable — I'll walk you through the operator's cockpit and show you the safety case, the savings, and the learning, in that order, ending with exactly what is and isn't real yet.

### What to say (talking points)

- Auth is token-based and fail-closed: the client verifies the token against the backend (GET /api/plans) BEFORE storing it or letting you in — a 401/403 is rejected, and even a backend outage (the fetch throwing) blocks entry rather than defaulting open.
- Two roles, enforced server-side: 'operator' can view AND run/create plans; only 'expert' can approve and deploy. The login field accepts either, but privilege separation is enforced by the FastAPI TokenAuth dependency on every protected route, not by hiding buttons in the UI.
- The secret is verified without being trusted prematurely: verifyToken does a read-only probe and explicitly does NOT mutate the stored TOKEN; only on a 200 does the handler call setToken to persist it. So a bad token never gets cached.
- This screen is the front of a layered safety architecture: door-level access control here, then the 3-net safety gate (k*sigma pre-tighten, robust CVaR_0.8 ensemble with safety ladder, zero-tolerance deploy backstop), then the hard inlet <= 26 C cap at every 15-min step. Governance and physics safety are separate, complementary layers.
- Honest scope: because deploy is currently shadow-mode and the live feed is simulated, the operator/expert split is the governance scaffold that will gate REAL actuation once a BACnet adapter + historian are connected — the access model is real even though actuation is not yet live.

**Money insight.** Safety in this system starts at the door: a fail-closed, role-separated token gate ensures that the dangerous capability — broadcasting 3 setpoints to 45 actuators in a ~2 MW hall — can never be exercised by an unauthenticated user or reached through a backend failure, with the expert role being the single human sign-off in front of deployment.

### State proactively (honest caveats)

- Authorization is enforced by the backend, not by this screen — the login form only verifies (a read-only GET /api/plans probe); the API checks role on every request. Don't oversell the UI as the security control.
- Tokens are static shared secrets (env-injected OPERATOR_TOKEN/EXPERT_TOKEN) stored in browser localStorage and sent as a plain Bearer header; this is a research-grade auth model, not production identity/expiry/rotation/TLS.
- There is a dev-only bypass: if DTWIN_INSECURE=1 the backend skips auth and treats every caller as expert. It must be OFF for any non-toy deployment; mention it proactively rather than letting a reviewer find it.
- Deploy runs in SHADOW MODE: the 45 commands an expert authorizes are recorded, never actuated. The role separation is real governance over the decision, but it does not yet gate a physical actuator.
- The telemetry/live feed elsewhere in the app is SIMULATED and labelled with a 'SIMULATED FEED' badge — the access layer is genuine even though the data it eventually guards is currently synthetic (a perturbed-plant EnergyPlus stand-in).
- No per-user identity or audit trail of WHO approved a plan beyond the role token (operator and expert are each a single shared secret) — worth flagging as a known gap for production.

### Likely questions & strong answers

- **Q: Where is the actual authorization enforced — surely not in this React form?**
  - **A:** Correct, the form is only a gate-keeper UI. Authorization is enforced server-side: every API call carries the bearer token (Authorization: Bearer …), and the FastAPI TokenAuth.require dependency checks the role on each route. The expert-only actions (approve, deploy) are wired as Depends(expert) and rejected with 403 ('requires expert role') even if a clever operator crafted the request by hand. The login screen just refuses to store a token that the backend won't honor for the minimum read scope (GET /api/plans).
- **Q: What does 'fail-closed' concretely mean here?**
  - **A:** Three things in the client, plus one in the backend. Client: (1) an invalid token makes verifyToken return false and we show 'Invalid token'; (2) if the backend is unreachable the fetch throws and we show "Can't reach the backend" — we do NOT fall through to an authenticated state; (3) the token is only persisted via setToken AFTER a successful verification, so a failed attempt leaves no cached credential. Backend: if no tokens are configured at all, TokenAuth returns 401 'auth not configured' rather than admitting anyone — unless the explicit dev opt-in DTWIN_INSECURE=1 is set, which I'll flag as a caveat. In every normal failure mode the default is 'no access'.
- **Q: Token-based auth is weak — what about expiry, rotation, transport security?**
  - **A:** Honest answer: in this prototype tokens are static shared secrets injected as env vars (OPERATOR_TOKEN/EXPERT_TOKEN), the field is masked, but the accepted token is stored in browser localStorage and sent as a plain Bearer header. There is also a dev bypass (DTWIN_INSECURE=1) that disables auth entirely. That is adequate for a single-origin research deployment behind the operator's own network, but for production I'd move to short-lived signed tokens (JWT/OAuth) with rotation, TLS-only transport, per-user identities for audit, and removal of the insecure bypass. The architecture already isolates verification in one function, so swapping the auth backend is a contained change.
- **Q: Why two roles and not finer-grained permissions?**
  - **A:** The risk axis that matters is 'can this person change the physical plant?'. Operator = observe and run/explore plans (no plant impact); expert = approve and deploy (potential plant impact). Two tiers map cleanly onto that single high-stakes boundary. Finer roles add complexity without adding safety for the one action that matters — deployment — which is also separately protected by the zero-tolerance deploy backstop and the hard inlet cap.
- **Q: Does the role separation actually protect anything today, given deploy is shadow-mode?**
  - **A:** Today it protects the integrity of the plan record and the deployment decision: only an expert can mark a plan approved/deployed, which is the human sign-off step. The physical actuation it would gate is currently recorded-not-actuated (shadow mode — 45 commands logged, never sent to a BACnet bus), so the governance is structurally complete but field-inert. When a real adapter and historian are wired in, this exact role boundary becomes the thing standing between a recommendation and a real setpoint change — no new access model required.

### Likely questions & strong answers

- **Q: Why beam search instead of a proper optimizer like Bayesian optimization or gradient methods?**
  - **A:** The objective is a black-box, expensive, noisy EnergyPlus simulation over only 3 bounded setpoints, with a hard safety constraint. Coarse-to-fine beam search is robust to that noise, trivially parallel across processes, and naturally respects box bounds — and a real E+ sensitivity sweep shows the landscape is dominated by one axis (airflow moves hall energy 15.5%, CHWST 3.4%, SAT ~0.5%), so a grid-then-refine search converges fast without gradients. We could layer Bayesian optimization on top later, but with 3 dims and a hard cap the marginal gain is modest versus the added fragility.
- **Q: Grid=5, beam=3, levels=3 — how many EnergyPlus runs is that, and is it enough to be near-optimal?**
  - **A:** Order-of-magnitude it is hundreds of full-week runs: up to 5^3=125 coarse candidates, then refined boxes around the top-3 over two more levels, with de-duplication of already-evaluated points. Each refinement geometrically shrinks the search box, so the final setpoints land at a fine sub-grid resolution within the original ranges. The well-posedness sweep shows the surface is smooth and airflow-dominated, so this budget lands near-optimal in practice; you can raise grid/levels for tighter resolution at more sim cost. I'd frame near-optimality as 'good engineering convergence on a smooth surface,' not a proven global optimum.
- **Q: The IT-load forecast looks almost flat — is your forecaster doing anything?**
  - **A:** It is doing the right thing: in this hall the 1F-2A IT load is genuinely near-constant in the historian, so a near-flat forecast is correct, not lazy. The forecaster is a week-aligned seasonal/persistence model; its real value is on weather, which is strongly diurnal and is the dominant energy driver here. Forecasting a flat series flat is the honest baseline — I'm not claiming a sophisticated NWP weather model.
- **Q: Does changing Workers change the answer?**
  - **A:** No. Workers only sets how many OS processes evaluate candidates in parallel, so it changes wall-clock time, not the result. We use processes rather than threads deliberately because the dctwin/EnergyPlus configuration is a process-global singleton and is not thread-safe; each worker gets its own process-isolated config.
- **Q: Where is the safety cap enforced — I don't see 26 C on this screen?**
